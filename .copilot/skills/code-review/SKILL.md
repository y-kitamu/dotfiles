# Code Review

## 🎯 目的

指定されたコード（ファイル・関数・差分）に対して、正確性・可読性・保守性・セキュリティ・パフォーマンスの観点から網羅的なレビューフィードバックを提供します。「機械が検出すべき問題（フォーマット・Lint）」には言及せず、人間ならではの洞察に集中します。

## 💎 コア原則

- **プロジェクト規約を最優先**: 既存コードのスタイル・パターンを把握してから指摘する
- **自動化できることは指摘しない**: フォーマット・Lint・型エラーはCIに委ねる
- **Nitを区別する**: 本質的な問題とスタイル上の好みを明確に分ける
- **良い部分も評価する**: 修正点だけでなく、優れた実装も言及する

## 📋 実行手順

### Step 1: コンテキスト把握

レビュー前に以下を確認する：
- コードの目的・変更の意図
- 関連するテストコードの有無
- 既存コードのスタイル・命名規則

### Step 2: 以下の観点でレビューを実施

#### 1. 正確性 (Correctness)
- ロジックは意図通りに動くか
- エッジケース（空配列、null/undefined/None、境界値、オーバーフロー）は考慮されているか
- エラーハンドリングは適切か（握りつぶし、伝播、リカバリ）
- 並行・非同期処理のバグ（競合状態、デッドロック、未awaitのPromise等）はないか

#### 2. 可読性 (Readability)
- 変数名・関数名・クラス名は意図を表現しているか
- 関数・メソッドは単一の役割に絞られているか（20〜30行を超える場合は要注意）
- 複雑なロジックに「なぜ」を説明するコメントがあるか
- マジックナンバー・マジックストリングは定数化されているか

#### 3. 保守性 (Maintainability)
- DRY原則（重複コードの排除）
- 変更の影響範囲は適切に局所化されているか
- 将来の変更を困難にするハードコーディングはないか
- 不要なコメントアウトや未使用コードはないか

#### 4. セキュリティ (Security — OWASP Top 10 基本確認)
- ユーザー入力が適切にバリデーション・サニタイズされているか
- SQLクエリ・シェルコマンドに文字列結合が使われていないか（インジェクション防止）
- 認証情報・シークレットがハードコードされていないか
- 認可チェック（権限確認）が漏れていないか
- エラーメッセージに内部情報（スタックトレース等）が露出していないか

#### 5. パフォーマンス (Performance)
- ループ内での明らかに非効率な処理（N+1クエリ、不要な再計算等）はないか
- 大量データに対して適切なページネーション・ストリーミングが使われているか
- メモリリークの可能性はないか（イベントリスナーの解除、リソースのクローズ等）

#### 6. テスト (Testing)
- 変更に対応するテストが存在するか
- テストはエッジケースをカバーしているか
- テストコード自体が可読・保守しやすいか

### Step 3: 結果の出力

以下のフォーマットで出力する：

```
## コードレビュー: [ファイル名 / 機能名]

### サマリー
[全体の印象・変更の妥当性の簡潔な評価 (2〜3文)]

### 指摘事項

**🔴 Critical（必須対応）**
- `[ファイル名:行番号]` **[観点]**: [問題点の説明]
  → 改善案: `[コードスニペットまたは説明]`

**🟡 Major（強く推奨）**
- `[ファイル名:行番号]` **[観点]**: [問題点の説明]
  → 改善案: ...

**🟢 Minor（改善余地あり）**
- `[ファイル名:行番号]` **[観点]**: [問題点の説明]

**💡 Nit（任意・スタイルの好み）**
- `[ファイル名:行番号]`: [提案]

### 良かった点
- [優れた実装・工夫されている点]

### テスト確認
[テストの充足度に関するコメント]
```

**優先度の基準**:
- 🔴 **Critical**: バグ・セキュリティ脆弱性・データ破損リスク → マージ前に必須対応
- 🟡 **Major**: 保守性・パフォーマンスへの重大な影響 → 強く推奨
- 🟢 **Minor**: 将来的な改善余地、可読性の向上 → できれば対応
- 💡 **Nit**: スタイル・個人の好み → 対応任意
