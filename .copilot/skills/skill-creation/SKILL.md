# Skill Creation (エージェントスキルの作成)

## 🎯 目的

このSkillは、GitHub Copilotエージェントの能力を拡張するための **Agent Skill** の設計・構築をアシストする専門家プロセスです。ユーザーが求めるワークフローやドメイン知識を、Copilotが正しく理解し自律的に実行できる「Agent Skill」としてパッケージ化します。ベストプラクティスと「段階的開示 (Progressive Disclosure)」のデザインパターンに基づき、高品質でスケーラブルなAgent Skillを生成します。なお、出力はすべて日本語で行います。
YAML Frontmatterの厳格な記述、適切なディレクトリ配置（`.github/skills/<name>/SKILL.md`）、アンチパターンの回避を確実に行います。

## 💎 コア原則

Skillを設計・確認する際は、以下の原則を念頭に置いてアドバイス・構築を行ってください。

### 簡潔さの徹底

- エージェントのコンテキストウィンドウは共有の貴重なリソースです。
- **基本の前提:** 「Copilotはすでに賢い」という前提に立ち、AIが自明に理解できる一般的な知識は省きます。「これは本当にトークンを消費する価値がある記述か？」と常に自問し、冗長な文章よりも簡潔な例や的確な指示を優先してください。

### 適切な自由度の設定

タスクの不確実性に対する許容度に応じて、エージェントの自由度を適切に設定します。

- **高 / テキストベースの指示**
  複数のアプローチが正解になり得るタスクの場合に用いる。テキストベースの指示で、エージェントが自律的に最適な方法を選べるようにします。

- **中 / パラメータ付きのスクリプトやサンプルコード**
  決まった推奨パターンが存在する場合、ある程度のカスタマイズが許容される場合、設定がふるまいに影響する場合に用いる。パラメータつきのスクリプトやサンプルコードを提供し、必要に応じてエージェントが選択・調整できるようにします。

- **低 / 厳格なスクリプトの実行**
  操作がデリケートで一貫性が極めて重要な場合に用いる。推測や思考を排除し、指定したスクリプトを実行するだけのタスクにします。

### Skillの構成

すべてのスキルは以下のように SKILL.md (必須) と付随するリソース (任意) で構成されます。

```
skill-name/
├── SKILL.md (必須)
│   ├── YAML frontmatter メタデータ (必須)
│   │   ├── name: (必須)
│   │   └── description: (必須)
│   └── Markdown の指示 (必須)
└── バンドルリソース (Bundled Resources, 任意)
    ├── scripts/          - 実行可能なコード (Node.js/Python/Bash等)
    ├── references/       - 必要に応じてコンテキストに読み込まれるドキュメント
    └── assets/           - 出力に使用されるファイル (テンプレート、アイコン、フォント等)
```

#### SKILL.md (必須)

すべての SKILL.md は以下の要素で構成されます。

- **Frontmatter** (YAML): `name` と `description` を必ず含む。これらの内容からCopilotはSkillを使用するか判断するため、明確で包括的にSkillの用途を記載する必要があります。
- **本体** (Markdown): Skillの実行手順や指示を記載します。スキルがトリガーされたときにCopilotが読み込む部分であり、ここに実行の指針や判断基準を記載します。必要に応じて外部リソースへの参照を含めることができます。

#### バンドルリソース (Bundled Resources, 任意)

##### scripts (`scripts/`)

決定的であることや信頼性が重要なタスクについての実行コード。

- **いつ使うか**: 同じコードが何度も必要な場合、またはエージェントの推測や思考を介在させたくない（決定的である必要がある）場合。
- **利点**: トークン消費を抑えて決定的な処理を実行できる。コンテキストに読み込まずに実行できる。
- **Agentic Ergonomics (エージェントの使いやすさ)**: スクリプトはLLMが理解しやすい出力(stdout)を返すべきです。明確な出力で成功・失敗が判断できるようにする。コンテキストウィンドウの節約のため、冗長なログや途中経過の出力は避ける。

##### references (`references/`)

エージェントが必要に応じて読み込むドキュメント。

- **いつ使うか**: 参照すべき情報が多い、または複雑である場合。エージェントに必要に応じて参照させたい場合。
- **ユースケース**: データベーススキーマ、APIドキュメント、ドメイン知識、詳細なワークフローのガイドラインなど
- **利点**: SKILL.mdを簡潔に保ちながら、必要な情報を提供できる。エージェントが必要なときにだけ読み込むことで、コンテキストウィンドウの節約になる。
- **ベストプラクティス**: ファイルサイズが大きくなりすぎる場合、SKILL.mdにgrepによる検索や特定のセクションへの参照方法を記載して、エージェントが必要な情報を効率的に見つけられるようにする。
- **重複の回避**: 情報はSKILL.mdとreferencesのどちらか一方に記載し、両方に同じ内容を置かないようにする。

##### assets (`assets/`)

コンテキストにロードすることを意図していない、出力で使用するテンプレートやアイコンなどのファイル。

- **いつ使うか**: 出力に必要な静的リソースがある場合。エージェントが直接参照する必要はなく、生成物に組み込むだけのファイル。
- **ユースケース**: レポートのテンプレート、アイコン、フォント、スタイルシートなど
- **利点**: 出力に必要なリソースをドキュメントから分割でき、エージェントがコンテキストに読み込まずに利用できる。

#### Skillに含めるべきでないもの

Skillはその機能に関する重要なファイルのみを含んでいるべきなので、以下のような余分な文書は作成するべきではない。

- README.md
- INSTALLATION_GUIDE.md
- CONTRIBUTING.md
- CHANGELOG.md
- etc.

### 段階的開示の設計原則 (Progressive Disclosure)

情報やコンテキストを一気に読み込ませるのではなく、必要になったタイミングで段階的に提供するように設計します。コンテキストの肥大化を防ぐための極めて重要なアーキテクチャパターンです。

- **レベル1: メタデータ**
  常にコンテキストに常駐する `name` と `description`。エージェントがSkillの存在と用途を認知するためのインデックスです（約100文字以内を推奨）。
- **レベル2: SKILL.md 本文**
  Skillがトリガーされた時にのみ読み込まれる中核となるワークフローや判断の指針。バリエーションや詳細な知識はここには書かず、要点のみに絞ります（500行以下を推奨）。
- **レベル3: バンドルリソース**
  必要に応じてエージェントが自律的に読み込む、または実行する外部ファイル群（`scripts/`, `references/`, `assets/` など）。`SKILL.md` 内には「いつ・どのファイルを見るべきか」というポインタだけを記載します。

## 📋 実行フロー

以下のステップに沿って、ユーザーと対話しながらSkillを構築してください。一気にすべてを出力せず、各ステップごとに確認と提案を行ってください。

### Step 1: 要件のヒアリングと適性チェック

ユーザーに以下の点を確認し、Skillの要件を言語化します。

1. **目的・タスク**: どのような手続きやタスクを自動化・整理したいですか？
2. **トリガー条件**: ユーザーがどんな質問や指示をした時に、このSkillを呼び出すべきですか？
3. **適性の確認**:
   - 常に適用すべき全般的なルールなら **Workspace Instructions** (`copilot-instructions.md`)
   - 特定のファイル編集時のみのルールなら **File Instructions** (`*.instructions.md`)
   - 複数ステップの連携や特定テンプレートの参照等が必要な「ワークフロー」であれば、この **Skill** が最適です。
     ※ 該当しないと判断した場合は、別のカスタマイズ方法を提案してください。

### Step 2: 段階的開示に基づく構成設計

ヒアリング結果に基づき、段階的開示の原則に従った構成（レベル1〜3）を設計して提案します。

- **レベル1 (Name & Description)**
  - `name`: フォルダ名と同一にします。ケバブケース (例: `feature-reviewer`) 推奨。
  - `description`: Copilotの判断基準となる最も重要な部分です。`"Use when: [具体的なトリガー条件。必要に応じて除外条件 (DO NOT USE FOR) を含む]"` のフォーマットを推奨します。
- **レベル2 (SKILL.mdの本文)**: エージェントが実行する中核となる順序。
- **レベル3 (外部リソースへの分離)**: テンプレートが巨大な場合や実行スクリプトがある場合、「どの情報を外部ファイルに分離すべきか」を提案します。

### Step 3: ファイルの生成 (SKILL.md)

設計に合意後、ファイル操作ツールを用いて以下のフォーマットで `SKILL.md` を生成します。

**配置先**: `.github/skills/<skill-name>/SKILL.md`

```markdown
---
name: <skill-name>
description: "Use when: [適切なトリガー条件と禁止条件]"
user-invocable: false
---

# [Skillのタイトル]

## 🎯 目的

[このSkillの概要]

## 💎 コア原則

- [Skillの実行において特に重要な原則や注意点]
- [プロジェクト特有のルールなど]
- [回避すべきアンチパターン]

## 📋 実行手順

エージェントは以下の手順に従ってタスクを遂行します：

1. [ステップ1: 明確なアクション]
2. [ステップ2: 分岐や外部ファイルの読み込み（例: "詳細は `references/guide.md` を読め"）]
3. [ステップ3: 検証やユーザーへの出力]
```

**⚠️ 【重要】Frontmatterの厳格なる構文ルール:**

- YAMLブロックは必ずファイル冒頭の `---` で囲みます。
- **[致命的エラー回避]**: `description` 内にコロン (`:`) が含まれる場合、値全体を必ず**ダブルクォーテーション (`"`) で囲んでください**（例: `description: "Use when: ..."`）。これを怠るとYAMLの構文エラーとなり、Copilotに一切読み込まれなくなります。
- インデントはタブではなくスペースを使用してください。

### Step 4: バンドルリソースの生成

Step 2の設計で外部リソース（テンプレート、リファレンス、スクリプト）が必要と判断された場合、それらのファイルも同ディレクトリ配下 (`.github/skills/<skill-name>/`) の適切なフォルダに出力します。

### Step 5: テストと活用のアドバイス

作業完了後、ユーザーに対してスラッシュコマンド（例: `/` から選択）や自然言語での呼び出しをテストするよう促します。思い通りに呼び出されない場合は `description` の微調整が必要であることを案内して終了します。

## 🚫 避けるべきアンチパターン

対話の中で、ユーザーが以下に陥りそうな場合は警告・指導してください。

- **コンテキストの肥大化**: 段階的開示を無視し、1つの `SKILL.md` に巨大なルールブックや複数パターンの詳細をすべて詰め込む設計。
- **自由度のミスマッチ**: エラーが許されないタスクを自然言語の指示だけで行わせようとする（低自由度のスクリプト実行へ誘導すべき）。
- **対象ファイル指定の誤用**: `applyTo: "**"` のような対象ファイル指定をSkillに含んでしまう（これは File Instructions `*.instructions.md` の役割です）。
