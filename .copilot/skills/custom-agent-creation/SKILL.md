# Custom Agent Creation (カスタムエージェントの作成)

## 🎯 目的

ユーザーの要件に基づき、GitHub Copilotの高品質なカスタムエージェント（`.agent.md`）を設計・生成します。単一責任の原則、最小権限の原則、および明確なハンドオフ契約に基づく相互連携（自律ワークフロー）を構築することを目的とします。なお、出力はすべて日本語で行います。

## 💎 コア原則

カスタムエージェントを設計・生成する際は、以下の原則を念頭に置いてアドバイス・構築を行ってください。

### カスタムエージェントのアーキテクチャ設計

- **単一責任**: 万能エージェントではなく、特定のタスクに特化したペルソナ（クリエイター、レビュアー等）として設計する。
- **オーケストレーターとワーカーの分離**: 複雑なタスクでは、ユーザーの曖昧な指示を受け取り適切なサブエージェントに振り分ける「オーケストレーター（親）」と、指示に従って作業のみを行う「ワーカー（子）」に分ける設計を推奨する。
- **Agent Skillとの役割分担**: エージェントの定義には「役割とツールの制限」を主に記載し、具体的な手順などは Agent Skill に記載する。
- **最小権限**: エージェントが使用できるツールはタスクの完遂に必要な最小限に絞り、権限の越権行為（例: レビュアーによる直接のコード改変）をシステムレベルで防ぐ。

### プロンプト設計

- **ペルソナと任務 (Responsibilities) の明確化**: 「あなたは厳格なセキュリティ専門家です」「あなたの責務は要件定義を作ることです」と、役割とゴールを明確に言語化する。特定のタスクのプロフェッショナルであることを強調し、エージェントがその役割に没入できるようにする。
- **明確な制約**: 「〜してはいけない」という否定形の制約を強く明示し、AIの推測や暴走を制御する。
- **必要な知識の指定**: エージェントが利用すべき Agent Skill や、参照すべきドキュメントを明示する。

## 📋 実行手順

エージェントは以下のステップに従ってタスクを遂行します：

### Step 1: 要件ヒアリング

ユーザーに以下の点を確認し、エージェントの要件を言語化する。

- **目的**: どのようなエージェントを定義して、何をエージェントにやらせたいですか？
- **エージェントの構成**: 単一エージェントで完結させたいですか？それとも複数のエージェントが連携する構成を想定していますか？（例: クリエイターとレビュアーの2エージェント構成）
- **定義済みのSkillの活用**: 既存の Agent Skill（例: `fxea-codebase-facts`）を活用する必要がありますか？必要な場合はどのSkillをどのように使いたいか教えてください。
- **他のエージェントとの連携有無**: 他のエージェントと連携する必要がありますか？必要な場合はどのエージェントとどのように連携させたいか教えてください。

### Step 2: 設計・設定の提案

ヒアリング結果に基づき、カスタムエージェントを設計してユーザーに提案する。

- **エージェントの構成案**: 単一エージェントで完結させるのか、複数エージェントが連携する構成にするのかを提案する。また、Copilot のベストプラクティスに従い、 Agent Skill として切り出したほうが良い内容があれば提案する。
- **各エージェントの詳細**: YAML Frontmatter の構成（`name`, `description`, `tools`, `handoffs`, `agents` 等）と、マークダウン本文のアウトラインを提案する。特に `description` の内容をもとにこのカスタムエージェントが呼び出されるか判断されるので、「発火条件」を取り入れた明確で簡潔かつ包括的な内容へとチューニングする。
- **他エージェントやエージェント間の連携方法**: エージェント間の連携の有無や受け渡す生成物などのワークフローを提案する。

### Step 3: カスタムエージェントの定義ファイル作成

- 提案に合意が得られたら、`.github/agents/<agent-name>.agent.md` にファイルを生成する。
- 複数エージェント構成の場合は、全てのエージェントのファイルを生成する。
- エージェントを生成する際はこの構造をベースにし、要件に合わせてカスタマイズしてください。

```markdown
---
name: "agent-name" # ケバブケース推奨
description: "Use when: [具体的な発火条件]。DO NOT use for: [除外条件]"
user-invocable: false
user-invocable: true # サブエージェント専用（裏方）にするなら false
tools: ["read", "search", "edit_file", "runSubagent"] # 必ず必要最小限に絞る
agents: ["next-agent"] # runSubagentで呼び出し可能な対象を制限（不要なら削除）
handoffs: # 不要なら削除
  - label: "Review Code"
    agent: "Reviewer"
    prompt: "以下の実装をレビューしてください。"
    send: true
---

# [Agent name]

あなたは〇〇な性格を持った〇〇の専門家（ペルソナ）です。あなたのミッションは〜〜です。

## 🎯 責務 (Responsibilities)

- [タスク1: 具体的に何を達成すべきか]
- [タスク2: どのような観点で作業を行うか]

## 🚫 制約事項 (Constraints)

- [例: コードの直接編集は行わず、指摘のみに留めること]
- [例: 推測で行動したり、架空の機能に依存しないこと]

## 📝 必要な知識 (Required Skills)

- [例: `fxea-codebase-facts` スキルを活用してコンテキストを把握すること]

## 👣 実行ステップ (Approach)

1. [最初のステップ: どのツールを使って初期情報を集めるか]
2. [思考・作業ステップ: どのような基準で評価・作成するか]
3. [完了ステップ: 結果をどのようなフォーマットで出力し、次にどう渡すか]
```

**⚠️ 生成時の注意点**

- **YAMLの構文**: `description` 内にコロン（`:`）などの特殊文字が含まれる場合、必ず値全体をダブルクォーテーション `"` で囲むこと。これができていないとCopilotに読み込まれません。
- **ステップバイステップ**: `実行ステップ` は人間ではなくAIに向けられた指示であるため、「考え方」ではなく「具体的なアクションの連続」として番号付きリストで明記してください。

### Step 4: ネクストアクション (Agent Skill作成) の提案

新たな Agent Skill の作成が必要な要件である場合、本スキル内で直接 SKILL.md を生成しようとしないでください（単一責任の原則）。
代わりに、.agent.md の生成が完了した後に、ユーザーに対して .agent.md の内容を踏まえた上で新たにスキル作成のタスクを立ち上げるよう提案してください。
提案例: 「エージェントの定義が完了しました。次に、このエージェントが使用する〇〇のスキルの作成を行ってください。」
