# ソフトウェア設計・実装の基本原則とルール

## 🎯 目的

本スキルは、設計やコーディングを行う際に遵守すべき基本的な原則とベストプラクティスを定義したものです。Copilot（エージェント）が保守性が高く、読みやすく、シンプルなコードを提案・生成するための基準を提供します。

## 💎 コア原則

以下の原則を常に念頭に置いてコードの生成やリファクタリングを行ってください。

### 1. 基本的な設計原則

- **KISS (Keep It Simple, Stupid)**: 最もシンプルな解決策を選択する。不必要な複雑さを避け、誰が見ても分かりやすいコードを書く。
- **YAGNI (You Aren't Gonna Need It)**: 今必要な機能だけを実装する。推測に基づく機能追加や汎用化は行わない。
- **DRY (Don't Repeat Yourself)**: 情報やロジックの重複を避ける。ただし、偶然の一致による不要な共通化（間違った抽象化）には注意する。

### 2. オブジェクト指向・アーキテクチャ設計

- **SOLID原則**:
  - 単一責任の原則 (SRP): 1つのモジュールは変更する理由が1つのみであるべき。
  - オープン・クローズドの原則 (OCP): 拡張に対して開いており、修正に対して閉じているべき。
  - リスコフの置換原則 (LSP): 派生型はその基本型と置換可能でなければならない。
  - インターフェース分離の原則 (ISP): 利用しないメソッドに依存することを強制しない。
  - 依存性逆転の原則 (DIP): 詳細ではなく抽象に依存するべき。
- **関心の分離 (SoC)**: ソフトウェアを独自の関心事を持つ別々の単位（レイヤー、モジュール）に分割する。

### 3. コーディングとリファクタリングのルール

- **ボイスカウトの規則**: コードに変更を加える際は、周辺のリファクタリングも行い、少しだけ綺麗にしてからコミットする。
- **Clean Code**: 変数や関数は意図が明確に伝わる名前にし、マジックナンバーは排除して定数化する。
- **関数のルール**: 1つのことだけを行い、なるべく短く保ち、副作用を避ける。引数は最小限にする。

### 4. エラー処理と防御的プログラミング

- **早期リターン (Guard Clauses)**: ネストを浅く保ち、異常系や前提条件のチェックは関数の先頭で行いすぐに返す。
- **フェイルファスト (Fail Fast)**: エラーが発生する可能性のある状況に陥った場合、処理を継続させず可能な限り早く失敗（例外のスローなど）させる。

### 5. ドキュメンテーション

- **不要なドキュメント作成の回避**: コード自体が自己文書化されるようにし、冗長なドキュメント作成を避ける。
